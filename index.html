General

  • Always check the browser requirement from client.
  • Set your editor to use soft tabs, 2 spaces.
  • Use Unix line endings.
  • On development, use as many white spaces and comments.
  • Always use sRGB colour profile when exporting images.
  • Optimise images with ImageOptim UI or CLI.
  • FontAwesome provides small and scalable icons.
  • Include CSS and JS at the end of the page, except when required otherwise.
  • Test performance using Google Page Speed and ySlow.
  • Validate your HTML or use the Chrome or Firefox extension.
  • Validate your CSS.

HTML

  • Your HTML should look reasonably good even with CSS (and JS) turned off.
  • Semantic markup helps.
  • When closing tags for layout, comment appropriately: </div> <!-- #header-->
  • Images should have alt attribute.
  • All form fields should have tabindex.
  • On forms, let CSS handle the sizing. Don't set input size.
  • On forms, use <label for="id-of-the-target-input">.
  • During development, it's fine to have multiple CSS files but they should be combined and minified on production.
  • Use <meta http-equiv="X-UA-Compatible" content="IE=edge">. This tells IE to use the latest version possible, not to fall back to compatibility.
  • Include CSS on top, JS on the bottom.
  • Emmet is our friend.

CSS

  • Let CSS handle capitalisation of titles.
  • CSS statements should never be inline.
  • CSS files should be included using <link> tag, not the import.
  • When not using any frameworks, use normalize.css
  • Group CSS by: normalize, global styles, layout styles, page styles.
  • Global styles should be declared using low priority selectors. E.g. h1 instead of body h1.
  • When not using any frameworks, size using REM. Browser support: Safari 5, Chrome, Firefox 3.6+, Opera11 and even Internet Explorer 9.
  • When not using any frameworks, use CSS browser prefixes.
  • Avoid using * selector (this is why we use normalize.css).
  • CSS ID should always be unique, CSS classes can be repeated on HTML.
  • Understand how to write efficient CSS.
  • Be careful when writing specific CSS selectors, you may need to override them. A good read.
  • Don't do div#something or div.something, only use #something or .something because it's faster.
  • Understand CSS box model, and box-sizing: border-box.
  • Alphabetise your CSS declarations please.
  • Name your IDs and classes what it is instead of what it looks like.
  • Use CSS shorthands.
  • Use sprites.
  • Use @font-face with the bulletproof syntax. We require .eot, .woff, .ttf, and SVG formats. Online converters are your friend http://www.fontsquirrel.com/tools/webfont-generator http://everythingfonts.com/font-face
  • Use SASS and Compass.

Javascript

  • Use feature detection instead of browser string detection. A good read.
  • Variable names should be meaningful.
  • JS should not be used for layouts. Minimise DOM manipulation.

Print

  • Include the CSS for print after all the other styles, we aim to override them all. <link rel="stylesheet" media="print" href="print.css">.
  • Remove unnecessary stuff with display: none. E.g. navigation, forms, anything the user won't be able to interact with.
  • Multi-column content should be put inside one.
  • Font should be (even more) readable.
  • CMS should prepare footnotes, otherwise we can use this.
